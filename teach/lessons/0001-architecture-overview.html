How the LLM Judge Works End-to-End

You built this. Now let's understand every piece of it.

The one-sentence summary

The LLM Judge is a daily batch service that reads prompt logs from cloud storage, sends each one to an LLM for scoring against a rubric, and writes a verdict back to storage — then sends an alert digest summarising anything that scored below a threshold.

The data flow

Every run follows this path, top to bottom:

Storage (production bucket) │ logs/{case_id}/{filename}.json ▼ StorageClient ← list_logs_modified_since() │ yields BlobMeta (case_id, filename, modified_time) ▼ runner.py ← pre-filter: skip existing verdicts, skip unmapped filenames │ for each remaining blob: ├─ rubrics/registry → get_rubric_name(filename) ├─ StorageClient → read_log(case_id, filename) → raw dict ├─ log_parser → parse_log() → ParsedTurn ├─ golden_examples → load_golden_examples() ├─ evaluator → evaluate_turn() → Verdict └─ StorageClient → write_verdict() │ ▼ alerts.py ← build digest of flagged verdicts → webhook / email │ ▼ RunSummary written to storage (runs/{run_id}.json)

The modules

There are 12 source files you need to know. Here they are by layer:

File Layer What it does
config.py config Reads ALL configuration from environment variables. One Settings object, injected everywhere.
models.py data Defines the data shapes: BlobMeta, ParsedTurn, Verdict, FlaggedItem, RunSummary.
log_parser.py logic Detects which of 4 log schemas a file uses and normalises it into a ParsedTurn.
rubrics/registry.py logic Maps log filenames to rubric names using regex patterns. First match wins.
rubrics/prompts/*.md logic The actual scoring criteria — one markdown file per rubric family (e.g. final_summary.md).
golden_examples.py logic Loads up to 2 reference "ideal" examples from storage for a given rubric. Used to calibrate the judge.
llm_client.py I/O Thin wrapper around the Anthropic SDK. Sends a system + user message, returns the raw text response.
evaluator.py logic Builds the judge prompt, calls LLMClient, parses the JSON response into a Verdict. Retries once on parse failure.
storage/base.py storage Abstract interface: list_files, read_file, write_file, file_exists. Every backend implements this.
storage/factory.py storage Reads STORAGE_PROVIDER from config and returns the right backend (gcs / s3 / azure / local).
storage/client.py storage High-level judge operations (read_log, write_verdict, etc.) built on top of three StorageProvider instances: prod, verdict, golden.
runner.py orchestration The "main loop". Lists blobs → filters → caps → evaluates in parallel (ThreadPoolExecutor) → writes verdicts → returns RunSummary.
alerts.py I/O Sends a daily digest of flagged verdicts via webhook (Slack/Teams) and/or SMTP email.
cli.py I/O Command-line entry point. Parses args, loads Settings, calls run_batch().
logging_setup.py config Configures structured JSON logging to stdout (for Cloud Run / log aggregators).

The storage abstraction layer

There are three storage layers worth distinguishing:

  • storage/base.py — the abstract interface (an ABC with 4 methods)
  • storage/gcs.py, s3.py, azure.py, local.py — the concrete backends
  • storage/client.py — the judge-specific façade on top: it holds three providers (prod, verdict, golden) and exposes domain operations

Why this design? The judge doesn't care whether it's talking to GCS, S3, or a local folder. Any module that needs to read/write only touches StorageClient — never a cloud SDK directly.

The four log schemas

Production writes logs in one of four formats. log_parser.py auto-detects which by inspecting the JSON keys — detection happens in this exact priority order:

SchemaKey signature (detection rule)Real example
duplicate_checkhas input1 AND input2check_duplication_1<>2.json
tool_usehas prompt AND output is a dictfilter_diagnoses.json — prompt exists, output is {"diagnoses":[…]}
standardhas prompt AND output is a stringfinal_summary.json
boundary_checkno prompt key at all{input: "…", output: "yes"} — only input + output, no prompt

Important distinction: the schema variant (detected above) and the rubric name (from the registry) are independent. filter_diagnoses.json has schema variant tool_use AND rubric grouping_boundary — the names sound similar but come from two completely different systems.

All four are normalised into the same ParsedTurn shape before evaluation.

Key design decisions

A few things that may look odd until you understand the reason:

  • Read-only production bucket — the judge never writes back to production logs, only to a separate verdict bucket. Isolation by design.
  • Skip existing verdicts — idempotency: re-running the job doesn't re-evaluate or overwrite already-judged logs.
  • Retry once on parse error — if the LLM returns malformed JSON, evaluator.py sends a correction prompt and tries once more before marking parse_error=True.
  • ThreadPoolExecutor — evaluating 500 logs sequentially would be slow. Workers run evaluations in parallel, controlled by JUDGE_MAX_WORKERS.
