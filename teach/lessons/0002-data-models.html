The Five Data Shapes

Everything in this system is data flowing between modules. This lesson is about the shapes that data takes.

Why this matters

Every module in your project communicates by passing typed objects — not raw dicts, not strings. If you understand the five data models in models.py, you can read any function signature and instantly know what goes in, what comes out, and what can fail.

The five shapes, in the order they appear during a run:

  1. BlobMeta — a pointer to a log file in storage
  2. ParsedTurn — one normalised log entry, ready to evaluate
  3. Verdict — the judge's decision on one turn
  4. FlaggedItem — a condensed view of a bad verdict, for the alert digest
  5. RunSummary — the aggregate result of the whole batch

1. BlobMeta

The lightest object in the system. Created by StorageClient when it lists files, consumed by runner.py. It carries just enough to know what to fetch — nothing about the content yet.

BlobMeta namedtuple (not Pydantic)
fieldtypewhat it is
case_idstrThe patient/case folder, e.g. "abc-123"
filenamestrThe log file name, e.g. "filter_diagnoses.json"
modified_timedatetime | NoneWhen the file was last written — used to enforce the lookback window. None if the storage backend can't determine it.
Why a namedtuple, not Pydantic?
BlobMeta is created in very high volume (once per log file) and never serialised to JSON. A namedtuple has no validation overhead and is immutable. Pydantic is reserved for objects that get written to storage or need validation.

2. ParsedTurn

Created by log_parser.py from the raw JSON dict. This is the normalised, schema-agnostic view of one log entry. After this point, the rest of the system doesn't care which of the four schemas the original file used.

ParsedTurn Pydantic BaseModel
fieldtypewhat it is
promptstrThe system/instruction prompt. For boundary_check and duplicate_check schemas this is a synthetic placeholder — the original log had none.
inputstrThe user input fed to the production LLM. For duplicate_check, this is a JSON string containing both input1 and input2.
outputAnyWhat the production LLM returned. A string for standard and boundary_check; a dict for tool_use and duplicate_check.
schema_variantstrOne of: standard, tool_use, boundary_check, duplicate_check. Set by the parser.
case_idstrCarried forward from BlobMeta.
filenamestrCarried forward from BlobMeta.
Key insight: the prompt field always has a value — even for schemas that don't log a prompt. The parser synthesises placeholders like "[boundary check — no separate prompt logged]" so that evaluator.py never has to handle None.

3. Verdict

The most important object. Created by evaluator.py, written to storage as JSON by the runner. Every log file eventually becomes exactly one Verdict.

Verdict Pydantic BaseModel
fieldtypewhat it is
case_idstrCarried from ParsedTurn.
filenamestrCarried from ParsedTurn.
prompt_typestrThe rubric name (e.g. "grouping_boundary"). Note: this is the rubric name, not the schema variant — they are different.
schema_variantstrCarried from ParsedTurn — records which of the 4 schemas this log used.
scoreint | None1–5 from the LLM. None when parse_error=True — the LLM responded but the response couldn't be parsed.
per_criteriondictBreakdown of the score by rubric dimension. E.g. {"completeness": 4, "accuracy": 3}. Empty dict on parse error.
flaggedboolTrue when score <= JUDGE_SCORE_THRESHOLD. Computed in code — never asked of the LLM.
reasoningstrThe LLM's explanation for the score. Empty string on parse error.
parse_errorboolTrue when both LLM attempts returned unparseable JSON.
raw_responsestr | NoneFirst 500 chars of the LLM response — only stored on parse error for debugging. None on success.
evaluated_atdatetimeUTC timestamp set at the start of evaluate_turn().
run_idstrStamped by the runner after evaluate_turn() returns. Empty string in older verdicts.
Two failure modes — and they look different:
  • Parse error: score=None, parse_error=True, raw_response="…"
  • Low quality (flagged): score=2, flagged=True, parse_error=False
These are completely different situations. A parse error means the judge itself broke. A flagged verdict means the judge worked and found a problem.

4. FlaggedItem

A trimmed-down view of a flagged Verdict, assembled by runner.py and handed to alerts.py. The alert digest doesn't need the full verdict — just enough to identify the problem and give a hint.

FlaggedItem Pydantic BaseModel
fieldtypewhat it is
case_idstrWhich case.
filenamestrWhich log file.
prompt_typestrWhich rubric family flagged it.
scoreint | NoneThe score that triggered the flag.
reasoning_snippetstrFirst 200 characters of the verdict's reasoning — a preview, not the full text.

5. RunSummary

Created at the end of runner.py and written to gs://VERDICT_BUCKET/runs/{run_id}.json. This is what the Streamlit dashboard reads to show you the history of batch runs.

RunSummary Pydantic BaseModel
fieldtypewhat it is
run_idstrA UUID generated at the start of run_batch().
started_atdatetimeUTC timestamp when the run began.
finished_atdatetimeUTC timestamp when the run ended.
total_logsintAll log files found in the lookback window — before any filtering.
evaluatedintHow many were actually sent to the LLM.
skipped_existingintSkipped because a verdict already existed (idempotency).
unmappedintSkipped because no rubric matched the filename.
flaggedintCount of verdicts with flagged=True.
errorsintUnhandled exceptions — network failures, write failures, etc.
parse_errorsintVerdicts where both LLM attempts failed to produce valid JSON.
by_prompt_typedict[str, int]Count of evaluations per rubric, e.g. {"final_summary": 12, "grouping_boundary": 8}.
flagged_itemslist[FlaggedItem]The actual flagged verdicts embedded here for the alert digest.
The accounting identity: total_logs = evaluated + skipped_existing + unmapped (plus any items capped by JUDGE_MAX_TURNS_PER_RUN). If these don't add up, something was missed.

Self-check — 5 questions

No peeking at the cards above.

1. A verdict has parse_error=True. What is the value of flagged?

True — parse errors are always flagged False — parse errors are never flagged Depends on JUDGE_SCORE_THRESHOLD setting None — same as score on parse error

2. The run_id field on Verdict is set by which module?

evaluator.py — it generates and assigns the UUID llm_client.py — it tags responses with a run ID runner.py — it stamps the Verdict after evaluate_turn() returns models.py — the Field default_factory generates it

3. A boundary_check log has no prompt field in its JSON. What does ParsedTurn.prompt contain?

None — the field is left empty for this schema An empty string — the default Pydantic value A synthetic placeholder string injected by log_parser.py The filename — used as a proxy for the missing prompt

4. What is the difference between errors and parse_errors in RunSummary?

errors is LLM failures; parse_errors is storage failures parse_errors is unparseable LLM JSON; errors is unhandled exceptions They count the same thing, errors is the total of parse_errors errors is from reading logs; parse_errors is from writing verdicts

5. How is Verdict.flagged determined?

The LLM is asked to include a flagged field in its JSON response It is set by alerts.py when building the digest It is a Pydantic validator that checks per_criterion scores It is computed in Python: score <= JUDGE_SCORE_THRESHOLD
